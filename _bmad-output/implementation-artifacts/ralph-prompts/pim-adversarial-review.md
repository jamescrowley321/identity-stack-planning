# py-identity-model — Full Codebase Adversarial & Red Team Review

**Run this prompt in a FRESH Claude Code session with no prior conversation context.**
**Working directory:** `~/repos/auth/py-identity-model/`

You are performing an independent adversarial and red team review of py-identity-model, a production OIDC/OAuth2.0 Python library that handles JWT decoding, token validation, JWKS retrieval, discovery document parsing, and advanced OAuth2 flows (PKCE, DPoP, PAR, JAR, Device Auth, Token Exchange, Introspection, Revocation). It offers dual sync/async APIs and is used as the token validation foundation for a multi-tenant identity platform.

**Critical context:** This library is security infrastructure. Bugs here cascade into every consumer. The codebase was largely generated by AI-driven development loops, and prior code reviews were done within the same conversation context as the implementation — making them inherently biased and shallow. Your job is to find what they missed.

## Phase 2: Red Team Review — Security Attack Surface

You are an attacker targeting applications that use this library. Your goal: find ways to bypass authentication, forge tokens, leak secrets, cause denial of service, or exploit protocol-level weaknesses.

### 2a: JWT Validation Attack Surface

Read and attack these files:

- `src/py_identity_model/core/token_validation_logic.py` — JWT validation orchestration
- `src/py_identity_model/core/jwt_helpers.py` — JWT decoding helpers
- `src/py_identity_model/core/validators.py` — Input/protocol validation
- `src/py_identity_model/sync/token_validation.py` + `aio/token_validation.py` — API wrappers

Look for:
- **Algorithm confusion attacks** — Can `alg: none` bypass validation? Can RS256 be downgraded to HS256 using the public key as HMAC secret? Is the algorithm allowlist enforced before signature verification?
- **Key confusion** — If multiple keys exist in JWKS, is `kid` matching strict? What happens with a missing `kid`? Can an attacker inject a JWK in the token header (`jku`/`x5u` injection)?
- **Claim validation gaps** — Is `iss` validated against a fixed allowlist or is it attacker-controllable? Is `aud` validation enforced? Is `exp` checked with clock skew tolerance? Is `nbf` enforced? What happens with missing required claims?
- **Signature verification** — Is the signature verified before claims are parsed? Can partial verification be exploited? Are all key types (RSA, EC, OKP) handled correctly?
- **Type juggling** — What happens if `exp` is a string? If `aud` is a single string vs array? If claims are nested objects instead of primitives?

### 2b: OIDC Discovery & JWKS Attack Surface

Read and attack these files:

- `src/py_identity_model/core/discovery_logic.py` — Discovery document parsing
- `src/py_identity_model/core/jwks_logic.py` — JWKS parsing/validation
- `src/py_identity_model/sync/discovery.py` + `aio/discovery.py`
- `src/py_identity_model/sync/jwks.py` + `aio/jwks.py`

Look for:
- **SSRF via discovery** — Is the discovery URL validated? Can an attacker control the `issuer` to force the library to fetch from an attacker-controlled server? Is `jwks_uri` from the discovery document followed blindly?
- **Cache poisoning** — Discovery docs and JWKS are cached. Can the cache be poisoned with malicious data? Is there a TTL? Can an attacker trigger cache invalidation and serve poisoned responses during the window?
- **Malformed JWKS** — What happens with duplicate `kid` values? With unsupported key types? With malformed key material? With extremely large JWKS responses (DoS)?
- **Key rotation race** — During key rotation, is there a window where valid tokens are rejected or invalid tokens accepted?

### 2c: OAuth2 Protocol Flow Attacks

Read and attack the advanced protocol modules:

- `src/py_identity_model/core/` — Look at every `*_logic.py` file for: auth_code_pkce, dpop, fapi, jar, par, device_auth, token_exchange, introspection, revocation, refresh
- Corresponding `sync/` and `aio/` wrappers

Look for:
- **PKCE bypass** — Is `code_verifier` actually validated? Can the flow complete without PKCE when PKCE is required?
- **DPoP proof replay** — Is the `jti` claim in DPoP proofs checked for uniqueness? Is `ath` (access token hash) validated? Can a captured DPoP proof be replayed?
- **Token exchange confusion** — Can `subject_token_type` and `requested_token_type` be manipulated to exchange a low-privilege token for a high-privilege one?
- **PAR injection** — Is the `request_uri` validated for origin? Can an attacker inject a `request_uri` pointing to malicious parameters?
- **JAR manipulation** — Is the signed request object validated before parameters are extracted? Can unsigned claims override signed ones?
- **Device code brute force** — Is the `user_code` space large enough? Is rate limiting enforced on the polling endpoint?
- **Introspection/Revocation auth** — Are these endpoints properly authenticated? Can any client introspect/revoke any token?

### 2d: HTTP Client & Transport Security

Read and attack:

- `src/py_identity_model/sync/http_client.py` — Thread-local HTTP client
- `src/py_identity_model/aio/http_client.py` — Singleton async HTTP client
- `src/py_identity_model/ssl_config.py` — SSL configuration
- `src/py_identity_model/core/http_utils.py` — HTTP configuration

Look for:
- **Connection reuse across tenants** — In a multi-tenant context, can one tenant's HTTP connection state leak to another?
- **TLS verification bypass** — Can SSL verification be disabled via environment variables? Is there a code path that skips certificate validation?
- **Timeout DoS** — What happens if an IdP never responds? Is there a read timeout? Can an attacker hold connections open via slow responses?
- **Redirect following** — Does the HTTP client follow redirects? Can an attacker redirect JWKS/discovery fetches to a malicious server?

## Phase 3: Blind Code Review — Logic & Correctness

Read EVERY file in `src/py_identity_model/` (not tests). For each module, look for:

1. **Sync/async parity** — Compare each `sync/*.py` with its `aio/*.py` counterpart. Are there logic differences? Missing error handling in one but not the other? Different default parameters?
2. **Error path handling** — Trace every `try/except` block. Are exceptions caught too broadly? Are errors swallowed silently? Do error paths clean up resources (HTTP connections, file handles)?
3. **Resource lifecycle** — Are HTTP responses always closed, even in error paths? Are connections returned to the pool? Is `finally` used where needed?
4. **Thread safety (sync)** — Is `threading.local()` used correctly? Can global state be corrupted by concurrent access?
5. **Async safety** — Are locks used where needed? Can the singleton HTTP client be double-initialized? Are there `await` calls inside critical sections that could yield and cause races?
6. **Type safety** — Are `Optional` types always checked before access? Are dict lookups guarded against `KeyError`? Are list accesses guarded against `IndexError`?
7. **Backward compatibility** — Do public API signatures match between versions? Are deprecated functions properly handled?

## Phase 4: Edge Case Exhaustion

For every public function in the library, consider these inputs:

- `None` / missing values for every parameter
- Empty strings, empty dicts, empty lists
- Extremely long strings (100KB+)
- Unicode edge cases (null bytes, RTL characters, zero-width spaces)
- Negative numbers for numeric parameters
- Integer overflow for timestamps
- Expired tokens with `exp` in the far past
- Tokens with `exp` in the far future (year 9999)
- Tokens with `nbf` in the future (not yet valid)
- Malformed JSON in responses
- HTTP 429 / 503 / connection reset responses
- Partial JSON responses (truncated)
- Response bodies with incorrect Content-Type headers

## Rules of Engagement

1. **Read the actual code** — do not guess or assume based on file names. Open every file.
2. **Trace call chains** — follow function calls to their implementation. Many bugs hide at the boundary between modules.
3. **Verify before reporting** — check if a potential issue is actually mitigated elsewhere. False positives waste time.
4. **Be specific** — include file paths, line numbers, and code snippets. Vague concerns are worthless.
5. **Assume the worst** — this is security infrastructure. A "minor" bug here can be a critical vulnerability in consumers.
6. **Check dependencies** — look at how `PyJWT`, `httpx`, and `cryptography` are configured and called. Misuse of a secure library is still a vulnerability.
7. **No hallucinated findings** — if you can't point to actual code, don't report it. Every finding must reference real code in the repo.
