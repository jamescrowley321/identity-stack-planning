# Product Requirements Document — Infrastructure Secrets Pipeline

**Author:** James
**Date:** 2026-03-29

## Executive Summary

This PRD defines the requirements for an infrastructure secrets pipeline spanning two initiatives: **HCP Terraform remote state** (I1) and **Infisical secrets management** (I2). These form a coherent pipeline — not two independent systems — where Terraform provisions infrastructure and stores state remotely, Infisical centralizes secrets generated by or consumed by Terraform, and applications consume secrets via CLI injection with zero code changes.

The current state is fragile: Terraform state lives as plaintext JSON on disk (no locking, no encryption, no versioning), and secrets are scattered across `.env` files shared informally. This PRD eliminates both problems. HCP Terraform provides encrypted, versioned, locked remote state on the free tier. Infisical provides centralized secrets management with audit logging, folder-based organization, and CLI injection (`infisical run`) that replaces `.env` files without touching application code.

The two initiatives can start in parallel and converge at integration. HCP Terraform setup is a single file change plus `terraform init -migrate-state`. Infisical setup is cloud account creation, secret import, and swapping `env_file:` for `infisical run` in Docker Compose entrypoints. The integration layer — Terraform outputs stored in Infisical, applications consuming from Infisical — is the third epic that ties them together.

The bootstrap problem reduces to exactly 2 secrets: the Infisical Machine Identity `client_id` and `client_secret`. Everything else lives in Infisical.

### What Makes This Worth Doing

**Secrets in `.env` files do not scale, audit, version, or rotate.** Every new developer gets credentials via Slack. Every CI pipeline duplicates secrets as GitHub Actions secrets. Terraform state contains sensitive outputs in plaintext. This pipeline eliminates all of that: one source of truth (Infisical), encrypted state (HCP Terraform), and a 2-secret bootstrap that replaces N secrets spread across N locations. The zero-code-change constraint (CLI injection) means the existing FastAPI backend and Vite frontend require no modifications — only the process wrapper changes.

## Success Criteria

### User Success

- **[IS]** A developer cloning the repo can run the application without manually creating `.env` files — `infisical run` handles secret injection
- **[IS]** Switching between dev/staging/prod environments requires changing a single `--env` flag, not swapping `.env` files
- **[TFP]** Running `terraform plan` or `terraform apply` uses remote state with automatic locking — no risk of concurrent state corruption

### Business Success

- **Portfolio credibility:** The infrastructure story moves from "I use `.env` files" to "I use remote state with encrypted secrets management and audit logging"
- **Consulting readiness:** The secrets pipeline is a reusable pattern that applies to any client engagement
- **Zero operational cost:** Both HCP Terraform and Infisical Cloud free tiers are sufficient for this project's scale

### Technical Success

- **[IS]** Zero `.env` files required for application runtime — all secrets sourced from Infisical via CLI injection
- **[TFP]** Terraform state encrypted at rest, versioned, and locked — no local `.tfstate` files in the working directory
- **[IS]** Bootstrap secret count reduced to exactly 2 (Infisical Machine Identity `client_id` + `client_secret`)
- **[IS]** Docker Compose services start successfully with `infisical run` wrapping their entrypoints
- **[TFP]** `terraform plan` shows no drift after state migration — infrastructure unchanged, only storage location moved

## Functional Requirements

### Epic 1 — HCP Terraform Remote State Migration

- FR-1: Authenticate the Terraform CLI with HCP Terraform via `terraform login`, storing the API token in `~/.terraform.d/credentials.tfrc.json`
- FR-2: Configure the HCP Terraform workspace `identity-stack-dev` with execution mode set to **local** — runs execute on the developer's machine, state is stored remotely (note: the `cloud` block already exists in `infra/main.tf`)
- FR-3: Configure a variable set named "Descope Credentials" containing `DESCOPE_MANAGEMENT_KEY` as a sensitive environment variable, applied to the `identity-stack-dev` workspace
- FR-4: Migrate existing local state to HCP Terraform via `terraform init -migrate-state`, confirming the state copy prompt
- FR-5: Verify migration by running `terraform state list` (resources match pre-migration) and `terraform plan` (no drift detected — output: "No changes")
- FR-6: Back up the local `terraform.tfstate` file before migration, then remove local state files (`terraform.tfstate`, `terraform.tfstate.backup`) after successful verification
- FR-7: Update `.gitignore` to include `*.tfstate`, `*.tfstate.backup`, `.terraform/`, and `*.tfvars` (if containing secrets) — prevent accidental commit of local state artifacts

### Epic 2 — Infisical Secrets Management Setup

- FR-8: Create an Infisical Cloud account and project for the identity-stack (née identity-stack) application
- FR-9: Create three environments in the Infisical project: `dev`, `staging`, `prod`
- FR-10: Create folder structure within each environment: `/backend`, `/frontend`, `/infra` — secrets scoped by consuming service
- FR-11: Import existing secrets from `.env` files into the appropriate Infisical folders and environments:
  - `/infra`: `DESCOPE_PROJECT_ID`, `DESCOPE_MANAGEMENT_KEY`
  - `/backend`: `DESCOPE_PROJECT_ID`, `DESCOPE_MANAGEMENT_KEY`, `DATABASE_URL`
  - `/frontend`: `VITE_DESCOPE_PROJECT_ID`, `VITE_DESCOPE_BASE_URL`
- FR-12: Create a Machine Identity with Universal Auth login method, generating `client_id` and `client_secret` for programmatic access (CI/CD and Docker Compose)
- FR-13: Install the Infisical CLI on the development machine and verify authentication via `infisical login`
- FR-14: Update the `docker-compose.yml` backend service to use `infisical run --env=dev --path=/backend --` as the entrypoint wrapper, replacing `env_file: .env`
- FR-15: Update the frontend build process to use `infisical run --env=dev --path=/frontend -- npm run build` for build-time secret injection
- FR-16: Document the 2-secret bootstrap: only `INFISICAL_MACHINE_IDENTITY_CLIENT_ID` and `INFISICAL_MACHINE_IDENTITY_CLIENT_SECRET` are required as environment variables on the host or in CI — all other secrets are sourced from Infisical

### Epic 3 — TF to Infisical to App Bidirectional Secret Flow

- FR-17: Add the `infisical/infisical` Terraform provider to `infra/main.tf` with `required_providers` block, authenticated via Machine Identity credentials (`client_id` + `client_secret`)
- FR-18: Create `infisical_secret` resources in Terraform for outputs that should flow to Infisical — at minimum, access key cleartext values generated by `descope_access_key` resources
- FR-19: Verify the bidirectional flow end-to-end: Terraform provisions Descope resources → sensitive outputs written to Infisical → backend application reads them via `infisical run` at startup
- FR-20: Ensure Infisical Machine Identity credentials for the Terraform provider are stored as sensitive variables in the HCP Terraform variable set (not in `.tfvars` files or git)
- FR-21: Create a `data "infisical_secrets"` data source to read secrets from Infisical into Terraform runs where needed (e.g., reading `DESCOPE_MANAGEMENT_KEY` from Infisical instead of from HCP Terraform variable sets — enabling Infisical as the single source of truth)
- FR-22: Tag all `infisical_secret` resources with metadata indicating their origin (`source = "terraform"`) to distinguish Terraform-managed secrets from manually created ones

## Non-Functional Requirements

### Security

- NFR-1: **[TFP]** Terraform state encrypted at rest by HCP Terraform — no plaintext `.tfstate` files on disk after migration
- NFR-2: **[TFP]** Sensitive Terraform outputs (marked `sensitive = true`) redacted in HCP Terraform UI and plan/apply logs
- NFR-3: **[IS]** No secrets stored in git history, `.env` files committed to the repository, or Docker image layers
- NFR-4: **[IS]** Infisical Machine Identity credentials (`client_id` + `client_secret`) are the only secrets stored outside Infisical — in GitHub Actions secrets or host environment variables
- NFR-5: **[IS]** All secret access logged in Infisical audit log with identity, action, timestamp, and IP address

### Performance

- NFR-6: **[TFP]** `terraform plan` and `terraform apply` execution time not meaningfully increased by remote state (local execution mode means compute is local; only state read/write goes to HCP Terraform)
- NFR-7: **[IS]** Application startup time increased by < 3 seconds due to `infisical run` secret fetching (CLI fetches all secrets in a single API call before spawning the subprocess)

### Compatibility

- NFR-8: **[TFP]** Local execution mode preserves compatibility with the custom `terraform-provider-descope` fork — no dependency on the provider being published to the Terraform Registry
- NFR-9: **[IS]** CLI injection (`infisical run`) requires zero changes to application code — the backend reads `os.environ` identically whether secrets come from `.env` files or Infisical injection
- NFR-10: **[IS]** Existing Docker Compose deployment continues to work without Infisical by falling back to `env_file: .env` — Infisical is an enhancement, not a hard dependency for local development
- NFR-11: **[TFP]** HCP Terraform free tier limits (500 runs/month, 5 users, 1 concurrent run) are sufficient — project uses < 50 runs/month

### Operational

- NFR-12: **[TFP]** State locking is automatic — no manual lock/unlock required, no DynamoDB table to manage (unlike S3 backend)
- NFR-13: **[TFP]** State version history retained by HCP Terraform — rollback to any previous state version possible via the UI
- NFR-14: **[IS]** Secret versioning enabled — every change creates a new version with full history viewable in Infisical UI
- NFR-15: **[IS]** Infisical Cloud free tier limits (5 users, unlimited projects/secrets, limited audit log retention) are sufficient for this project's scale
